# REPO OPERATING MODEL

## Why This Exists

The original lightweight files in this repository are useful as scaffolding, but they are intentionally short.
The v2 and v3 files are the stronger operating documents intended for real use with large-context assistants.

Use the newer files when instruction fidelity, compliance, and quality matter more than compactness.

---

## Recommended Active Files

Use these as the active operating set:

- `AI_EXCELLENCE_V3.md`
- `LEVER_SYSTEM_V2.md`
- `ROLE_SYSTEM_V2.md`
- `ORCHESTRATION_V2.md`
- `.github/copilot-instructions.md`

The older files may remain in the repository as drafts, scaffolds, or comparison baselines.

---

## Interpretation Model

### 1. Doctrine
Doctrine defines:
- what behavior is globally required
- what is forbidden
- how tradeoffs are resolved
- what failure looks like

### 2. Levers
Levers define:
- how strongly the model behaves along a dimension
- whether the task should skew toward speed, rigor, autonomy, or caution

### 3. Roles
Roles define:
- who is best suited to the current dominant need
- what each behavior specialization should optimize for

### 4. Orchestration
Orchestration defines:
- who acts first
- who checks
- when to escalate
- when to retry
- when to stop

---

## Practical Use Rule

Do not treat these files as decorative philosophy.
Treat them as an operating system for behavior.

The intended result is:
- more proactive assistance
- stronger recommendations
- better quality control
- more human-like judgment
- less shallow compliance
- less repeated low-value clarification

---

## Maintenance Rule

When the system behaves poorly, diagnose the problem at the correct layer:
- wrong values or constraints → doctrine
- wrong intensity → levers
- wrong specialization → roles
- wrong routing or release behavior → orchestration

Improvement should happen at the correct layer rather than by randomly editing every file.
